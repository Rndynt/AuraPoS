# AuraPoS Critical Billing Entitlement Fix: New Tenants Must Never Auto-Upgrade to Growth/Pro

You are working on this repository:

https://github.com/Rndynt/AuraPoS

## Context

A serious entitlement bug exists in AuraPoS.

When a new tenant registers, the tenant can automatically become `plan_tier = growth` or even `plan_tier = pro` depending on the selected business type template. This is dangerous because a new tenant can receive paid-plan access without payment, invoice, billing webhook, or admin approval.

Current observed behavior from UI:

- New cafe/restaurant tenant shows active package: `Growth`
- Marketplace unlocks Growth modules like Table Management and Kitchen Display
- Registration for retail/laundry/service/PPOB can fail with:
  - `Required order types are not seeded: WALK_IN`
  - `Internal server error`

Existing code findings that must be verified in the repo before editing:

- `apps/api/src/services/registrationService.ts`
  - `registerTenantOwner()` inserts a tenant using `template.tenantDefaults.plan_tier`.
  - This means onboarding trusts business type templates for billing tier.
- `packages/application/tenants/businessTypeTemplates.ts`
  - `CAFE_RESTAURANT`, `RETAIL_MINIMARKET`, `LAUNDRY`, `SERVICE_APPOINTMENT` use `plan_tier: 'growth'`.
  - `DIGITAL_PPOB` uses `plan_tier: 'pro'`.
- `apps/api/src/http/controllers/TenantsController.ts`
  - `PATCH /api/tenants/plan` directly updates tenant `planTier`.
  - This endpoint currently allows switching plans from the app UI without billing authority.
- `apps/pos-terminal-web/src/pages/marketplace.tsx`
  - UI calls `PATCH /api/tenants/plan` directly when switching plans.
- `apps/api/src/seed.ts`
  - Seeds order types only `DINE_IN`, `TAKE_AWAY`, `DELIVERY`.
  - Does not seed `WALK_IN`.
  - Seeds business type codes like `LAUNDRY_SERVICE` and `SERVICE_BUSINESS`, while frontend/core uses `LAUNDRY` and `SERVICE_APPOINTMENT`.
- `packages/core/enums.ts`
  - Canonical business types are:
    - `CAFE_RESTAURANT`
    - `RETAIL_MINIMARKET`
    - `LAUNDRY`
    - `SERVICE_APPOINTMENT`
    - `DIGITAL_PPOB`
  - Canonical order type includes `WALK_IN`.

## Goal

Fix the entitlement model so registration is safe.

A newly registered tenant must always start on `free` / Starter plan unless a trusted backend-only billing/admin flow explicitly upgrades it.

Business type selection may configure workflows, starter catalog, order types, and recommended modules, but it must never grant paid plan access by itself.

## Non-Negotiable Security Rules

1. Never let a public registration request choose or inherit a paid `plan_tier`.
2. Never let browser/client UI directly upgrade a tenant plan.
3. `plan_tier` must be treated as billing entitlement, not a business-template preference.
4. Business type templates must not grant paid modules/features as active defaults.
5. Paid features/modules can be displayed as recommended/locked, but not active.
6. New tenant registration must be deterministic:
   - `plan_tier = free`
   - `subscription_status = active` or another safe non-paid status, but do not use Growth/Pro trial as entitlement.
7. `PATCH /api/tenants/plan` must be locked behind a trusted server-side authorization mechanism or disabled until billing exists.
8. Do not break existing POS basic registration flow.
9. Do not silently bypass errors. Return clear API errors.
10. Add tests or regression checks proving this cannot happen again.

## Required Fixes

### 1. Fix Business Type Templates

File:

- `packages/application/tenants/businessTypeTemplates.ts`

Change all business type templates so they do not assign paid plan tiers.

Every template must use:

    tenantDefaults: {
      plan_tier: 'free',
      subscription_status: 'active',
      settings: ...
    }

or use a safe non-paid status if the current system expects it, but `plan_tier` must always be `free`.

Important:

- Remove or rewrite comments that say cafe/retail/laundry starts in Growth trial.
- Do not keep any default `plan_tier: 'growth'` or `plan_tier: 'pro'` in business templates.
- `DIGITAL_PPOB` must also start as `free`, not `pro`.

### 2. Split Business Template Defaults from Paid Entitlements

Business templates currently include paid features such as:

- `kitchen_ticket`
- `kitchen_display`
- `kitchen_printer`
- `order_notifications`
- `inventory_tracking`
- `inventory_reports`
- `barcode_scanner`
- `label_printer`
- `analytics_dashboard`
- `payment_gateway`
- `api_integration`

These must not be inserted as active `tenant_features` for free tenants.

Refactor the templates so `features` only contains free-plan features allowed by `PLAN_FEATURE_MAP.free`.

Current free features from `apps/api/src/constants/planFeatureMap.ts` include:

- `product_variants`
- `partial_payment`
- `discounts`
- `order_queue`
- `receipt_printer`
- `sales_reports`

The final behavior must be:

- Cafe/restaurant new tenant:
  - free plan only
  - no active KDS / kitchen paid features
  - paid restaurant modules appear locked/recommended in Marketplace
- Retail new tenant:
  - free plan only
  - no active inventory advanced features unless they are explicitly part of free plan
- Laundry new tenant:
  - free plan only
  - no active label printer/order notification paid features
- Service/appointment tenant:
  - free plan only
  - no active appointment/analytics paid entitlement unless free plan explicitly allows it
- Digital/PPOB tenant:
  - free plan only
  - no payment gateway/api integration entitlement

If any module should be shown as recommended for a business type, store it as configuration/recommendation only, not as active entitlement.

### 3. Fix Module Defaults

Business templates currently enable paid modules by default, for example:

- Cafe:
  - `enable_table_management: true`
  - `enable_kitchen_ticket: true`
  - `enable_delivery: true`
- Retail:
  - `enable_inventory_advanced: true`

- Laundry:
  - `enable_loyalty: true`
  - `enable_delivery: true`
- Service:
  - `enable_appointments: true`
- PPOB:
  - `enable_multi_location: true`

These must not be enabled for free tenants if the module requires Growth/Pro.

Update module defaults so new tenants only have modules that are allowed under `free`.

Recommended safe default:

- `enableInventory` / `enable_inventory` may be true only if Marketplace treats basic stock as free/core.
- All Growth/Pro modules must default to false:
  - table management
  - kitchen ticket/KDS
  - loyalty
  - delivery if considered Growth
  - inventory advanced
  - appointments
  - multi location

Make sure the naming is correct between domain snake_case and DB camelCase mapping.

### 4. Enforce Registration Plan Safety in Backend

File:

- `apps/api/src/services/registrationService.ts`

Even after fixing templates, add a hard backend guard so this bug cannot reappear.

Inside `registerTenantOwner()`:

- Do not insert tenant with `template.tenantDefaults.plan_tier` directly unless it is guaranteed `free`.
- Prefer a constant:

    const DEFAULT_ONBOARDING_PLAN_TIER = 'free';

- Insert tenant with:

    planTier: DEFAULT_ONBOARDING_PLAN_TIER

- Keep subscription status safe:

    subscriptionStatus: 'active'

or use an explicit safe onboarding status if already supported.

Add guard:

- If any selected template tries to define a non-free `tenantDefaults.plan_tier`, registration must not grant that paid tier.
- Either ignore it and force free, or throw a clear developer error during tests.
- For production onboarding, force free is safer.

Also keep the existing safety guard that validates template features against plan map, but update it to validate against `free`, not template tier.

Expected validation:

    const allowedForOnboardingPlan = PLAN_FEATURE_MAP.free;
    const outOfPlan = featureCodes.filter(code => !allowedForOnboardingPlan.includes(code));
    if (outOfPlan.length > 0) throw RegistrationError(...)

This ensures registration cannot seed paid features.

### 5. Lock Plan Switching Endpoint

File:

- `apps/api/src/http/controllers/TenantsController.ts`

Current `PATCH /api/tenants/plan` is dangerous because it directly changes `planTier`.

Implement one of these safe options.

Preferred interim option until real billing exists:

- Keep endpoint for future internal billing use, but require a trusted internal secret header.
- Header name:

    x-internal-billing-secret

- Environment variable:

    BILLING_INTERNAL_SECRET

Rules:

- If `BILLING_INTERNAL_SECRET` is not configured, return 403 for all plan switching attempts.
- If header is missing or mismatched, return 403.
- Only after the internal secret is valid, allow plan tier update.

Response for unauthorized access:

    {
      "success": false,
      "error": "Plan changes are restricted to the billing/admin system.",
      "code": "BILLING_AUTH_REQUIRED"
    }

Do not allow normal tenant/browser session to upgrade plan.

Also make sure cache invalidation still works after authorized plan update.
